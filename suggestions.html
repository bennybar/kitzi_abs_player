Kitzi Review + Mockup

UI suggestions for the player, download state, and metadata reliability.

This is a review-only deliverable based on the current Flutter code in full_player_page.dart, download_button.dart, playback_repository.dart, and the book detail page. The goal is to make the full player feel tighter and more adaptive, keep download progress readable, and reduce narrator disappearance for offline books.

Review Findings

Layout Full player vertical rhythm is still too rigid.

The large center gap is caused by the screen distributing space by section, not by content priority. The artwork block, metadata block, waveform area, and progress area are stacked well, but their parent layout still behaves like fixed zones rather than a fluid composition.

Progress The player currently mixes “primary progress” and “chapter context” well in concept, but not yet in presentation.

The chapter map below the main bar is the right direction. It should behave like a compact context rail, not another heavy section with its own visual gravity.

Download The in-progress button should remain a single stable control.

A filled background with downloaded / total and percentage is the strongest pattern here. It keeps the footprint stable and avoids the awkward “button shrinks and a second progress bar appears” behavior.

Data Offline narrator loss was likely a metadata hydration gap.

The right fix is not a player-level fallback label. It is ensuring the offline warm-load path hydrates narrator consistently from local book metadata, cached server metadata, or both.

Suggested Full Player

The mockup below makes the upper area genuinely dynamic. When title, author, and narrator are short, the cover grows and the progress block moves upward. When metadata is long, the cover shrinks first. The main progress bar remains the user-selected primary mode, and the chapter rail sits below as lightweight context. Waveform is removed entirely from the proposed player.

Suggested Download States

Keep download as one stable button with zero height changes. The fill itself communicates progress. The text shows exact percentage plus downloaded amount over total amount. No second progress strip and no vertical stacking inside a constrained button.

Implementation Notes

These are the concrete code-level changes I would recommend when you want to move from review to implementation.

Full player layout: replace the current “large flexible top section plus fixed lower sections” mentality with a content-priority layout. The cover should be the elastic element. Metadata, progress, and controls should keep tight intrinsic spacing.
Chapter context: keep the existing primary progress behavior from settings, but always render the book-level chapter map below it in a lighter style. Add chapter name and chapter elapsed/total only once.
Download button: keep the active state to a single horizontal row. Avoid any inner vertical layout that can overflow within a fixed-height button.
Narrator reliability: make offline warm load follow the same metadata hydration rules as the online path, and prefer cached book DB values when server metadata is unavailable.
Settings cleanup: remove waveform animation as a feature, remove its settings toggle, and reclaim that space for cleaner progress and transport hierarchy.
